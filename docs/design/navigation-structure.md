## 2. ボトムタブ構成

### 2.1 タブ一覧

| タブ番号 | ラベル | アイコン（Lucide 系） | ルート | Web 対応 |
|---------|------|--------------------|-------|--------|
| 1 | ホーム | `Home` | `(tabs)/feed` | `/feed` |
| 2 | 検索 | `Search` | `(tabs)/search` | `/search` |
| 3 | 通知 | `Bell` | `(tabs)/notifications` | `/notifications` |
| 4 | プロフィール | `User` | `(tabs)/profile` | `/users/[自分のID]` |
| 5 | もっと見る | `MoreHorizontal` | `(tabs)/more` | Web 版の「もっと見る」メニューに相当 |

**Web 版との差異:** Web の MobileNav には「ホーム / 検索 / 通知 / メッセージ」の 4 タブ + 「もっと見る」があるが、モバイルアプリでは MVP スコープ外の機能（メッセージ等）を省き、代わりに「プロフィール」と「もっと見る」をタブとして設ける 5 タブ構成とする。

**「もっと見る」タブの役割:** Web 版の「もっと見る」メニューと同等の項目構成をフルスクリーンのメニュー画面として提供する。ネイティブ画面のない機能は expo-web-browser でアプリ内ブラウザを開く。詳細仕様は `docs/design/more-menu.md` を参照。

### 2.2 BottomTabBar 仕様

```
高さ: 60pt（セーフエリア下端は別途追加）
背景: colorNavBackground (#f7f7f7) / backdrop blur（不透明度 90%）
上端境界: 1pt solid colorBorderLight
アイコンサイズ: 20 x 20pt
タップターゲット: 幅は 1/5 画面幅（flex: 1）、高さは 60pt — 両軸とも 44pt 以上
ラベル: textXs（10pt） / letterSpacingWidest（1.5） / 上マージン 4pt
アクティブ色: colorNavIconActive（#252525）/ fontWeight: 700
非アクティブ色: colorNavIconInactive（#72727280）
アクティブインジケータ: アイコン上部の小円点 + アイコン下部の墨筆風短線（Web の ActiveBarIndicator と同じ意匠、React Native の SVG で再現）
通知バッジ: 通知タブのアイコン右上に赤いバッジ（未読数 or 点のみ）
```

**タップターゲット幅の変更:** 4 タブ時は `1/4 画面幅` だったが、5 タブ化により `1/5 画面幅` に変わる。375pt 幅の端末では 1 タブあたり 75pt となり、44pt 最小要件は引き続き満たす。

**iOS / Android 差異:**
- iOS は SafeAreaView で下端の home indicator エリアを自動確保する。
- Android では NavigationBar の高さを `react-native-safe-area-context` で取得してパディングを適用する。

## 3. ルートレイアウト（認証ガード）

`app/_layout.tsx` に認証状態の参照と画面振り分けを集約する（画面ごとの個別ガードは作らない）。

```
起動時の処理フロー:
1. expo-secure-store からトークンを確認（lib/auth/ 経由）
2. 未認証 → (auth)/login へ redirect
3. 認証済みが (auth) 配下 → (tabs)/feed へ redirect
4. ディープリンクで保護画面に到達 → ログイン後に元の画面へ復帰（lib/push/ の対応表を参照）
```

### 4.1 (auth) スタック

#### ログイン画面 `(auth)/login`

**概要:** メールアドレス + パスワードによるログイン。Google OAuth オプション。

**画面要素:**
```
[ヘッダー: アプリロゴ + 「BON-LOG」]
[見出し: 「ログイン」]
[メールアドレス入力]
[パスワード入力]
[ログインボタン（主要アクション）]
[区切り]
[Google でログイン]
[新規登録へのリンク]

- ローディング: ボタンをスピナーに変えて連打防止
- エラー: 「メールアドレスまたはパスワードが正しくありません」（インラインエラー）
- オフライン: 「接続を確認してください。ネットワークが利用できない状態です。」

### 4.2 (tabs) — ボトムタブ画面

#### フィード画面 `(tabs)/feed`

**概要:** フォロー中ユーザーの投稿が時系列で並ぶタイムライン。MVPのコア画面。

**画面要素:**
```
[ナビゲーションヘッダー]
  左: アプリロゴ（小）
  中央: 「ホーム」
  右: （将来: フィルター）

[フィードリスト（FlatList / 無限スクロール）]
  各 PostCard:
    [アバター（44pt）]
    [ユーザー名 + 日時]
    [投稿本文（最大 500/2000 文字）]
    [画像ギャラリー（1〜4枚）]
    [ジャンルタグ（最大 3 つ）]
    [アクションバー: いいね / コメント / リポスト / ブックマーク]

[FAB: 投稿ボタン（右下固定 / BottomTabBar 上 spacing4 + 60pt）]

[Pull-to-refresh: 引っ張って最新を取得]
```

**状態:**

| 状態 | 表示 |
|------|------|
| ローディング | PostCard スケルトン x 3 を縦並び |
| 空 | イラスト + 「まだ投稿がありません。フォローしてみましょう。」+ 検索へのリンク |
| エラー | 「タイムラインを読み込めませんでした。」+ 再読み込みボタン |
| オフライン | 画面上部にバナー「オフライン中です。接続が回復したら自動的に更新されます。」|

[グループ 2: メニュー]
  発見（アプリ内ブラウザ /explore ↗）

  ブックマーク（アプリ内ブラウザ /bookmarks ↗）

[グループ 3: 育成ガイド]
  農薬・病害虫（アプリ内ブラウザ /pesticides ↗）

### 4.3 タブ外スタック画面

#### 投稿詳細画面 `posts/[id]`

**概要:** 投稿本文・画像・アクションバー + コメントスレッドを表示。

**ナビゲーション:** タブスタックの上に Modal として積む（iOS: スライドアップシート or スタック遷移。Android: スタック遷移）。

**画面要素:**
```
[ナビゲーションヘッダー]
  左: 戻るボタン（iOS: "<" / Android: "←"）
  中央: 「投稿」

[投稿本文エリア（PostCard の disableNavigation=true 版）]

[コメント入力バー（画面下部 / キーボード上に追従）]
  [アバター（小）]
  [テキスト入力: 「コメントを追加...」]
  [送信ボタン]

[コメントリスト（FlatList / 無限スクロール）]
```

**iOS / Android 差異:**
- iOS: ヘッダーの戻るボタンは左端 "<" テキスト（デフォルト）、または「閉じる」（モーダル表示時）
- Android: ハードウェアバックボタンで戻る動作に対応する（Expo Router が処理）

**状態:**

| 状態 | 表示 |
|------|------|
| ローディング | 投稿スケルトン + コメントスケルトン x 2 |
| 投稿が存在しない | 「この投稿は見つかりません。」+ フィードへ戻るリンク |
| コメントなし（空） | 「まだコメントはありません。最初のコメントをしてみましょう。」|
| エラー | 「投稿を読み込めませんでした。」+ 再読み込みボタン |
| オフライン | オフラインバナー |

#### 他ユーザープロフィール画面 `users/[id]`

**概要:** 他ユーザーのプロフィール。フォロー・ブロック・通報アクションあり。

**自分のプロフィールとの差異:**
- 「プロフィール編集」の代わりに「フォロー / フォロー中」ボタン
- メニュー（⋮）から「ブロック」「通報」を提供（store-compliance.md 要件）

**状態:** 自分のプロフィールと同じ 4 状態 + 「非公開アカウント / フォローしていない」状態（投稿リストを非表示にする）

#### 新規投稿画面 `posts/new`

**概要:** テキスト・画像・ジャンルを入力して投稿を作成する。

**ナビゲーション:** フィード画面の FAB タップ → Modal として表示（iOS: ボトムシートまたはフルスクリーン。Android: フルスクリーン Modal）。

**画面要素:**
```
[ヘッダー]
  左: 「キャンセル」
  中央: 「新規投稿」
  右: 「投稿」ボタン（送信中は disabled + spinner）

[テキストエリア（自動拡縮）]
  プレースホルダー: 「盆栽の記録をシェアしましょう...」
  文字数カウンター（右下: 現在文字数 / 上限）

[画像エリア（横スクロール / 最大 4枚）]
  [+ 画像を追加ボタン]
  [各画像のサムネイル + 削除ボタン]

[ジャンル選択（最大 3 つ）]
  [タグリスト: 松柏類 / 雑木類 / 草もの / 用品・道具 / 施設・イベント / その他]

[文字数・画像数の上限表示]
  プレミアム: 2000文字 / 6枚
  無料: 500文字 / 4枚
```

**状態:**

| 状態 | 表示 |
|------|------|
| 送信中 | 投稿ボタンをスピナーに変更、全入力を disabled |
| 成功 | Modal を閉じてフィードに戻る。フィードを invalidate して最新表示 |
| エラー（400） | 入力エラーをインライン表示（文字数超過・禁止語等） |
| エラー（429） | 「本日の投稿上限に達しました。明日またお試しください。」|
| オフライン | 「投稿できません。ネットワーク接続を確認してください。」|

---

**アカウント設定画面 `settings/account`**

Google Play 審査要件（store-compliance.md）に基づき、**アカウント削除導線を必ず配置する。**

```
[メールアドレス変更]
[パスワード変更]
[---]
[危険ゾーン]
  [アカウントを削除する]（赤色テキスト）
```

アカウント削除タップ → 確認ダイアログ（2 ステップ確認）→ サーバー API 呼び出し → ログアウト処理（auth-tokens.md の手順に従う）。

---

**サブスクリプション画面 `settings/subscription`**

Google Play 審査要件に基づき、**購入の復元ボタンを必ず配置する。**
Stripe などの外部決済ページへの誘導は一切出さない（store-compliance.md）。

```
[現在のプラン表示]

## 5. ヘッダー・戻る挙動の方針

### 5.1 ヘッダー種別

| 種別 | 適用場面 | 左要素 | 中央 | 右要素 |
|------|---------|-------|------|-------|
| タブヘッダー | (tabs) 配下 | ロゴ（オプション） | 画面タイトル | アクション（任意） |
| スタックヘッダー | posts/[id]、users/[id]等 | 戻るボタン | 画面タイトル | アクション（任意） |
| モーダルヘッダー | 投稿作成・編集 | 「キャンセル」テキストボタン | 画面タイトル | 「投稿」テキストボタン |

### 5.2 戻る挙動の OS 差異

| OS | 操作 | 挙動 |
|----|-----|------|
| iOS | ヘッダー左 "<" タップ | スタックをポップ（右から左へスライドバック） |
| iOS | 画面左端からのスワイプ | インタラクティブポップジェスチャー |
| Android | ヘッダー左 "←" タップ | スタックをポップ |
| Android | システムバックボタン / ジェスチャー | Expo Router が処理（スタックポップ or ホームへ） |
| 両 OS | モーダル「キャンセル」 | 未保存内容がある場合: 破棄確認ダイアログを出してから閉じる |

### 5.3 タブバーの表示・非表示

- (tabs) 配下の画面: BottomTabBar を常時表示
- スタック画面（posts/[id] 等）: BottomTabBar を非表示にする（タブの上に full-screen でスタック）
- モーダル画面（投稿作成等）: BottomTabBar を非表示にする

## 6. 4 状態の文言トーン方針

すべての画面にローディング / 空 / エラー / オフラインの 4 状態を定義する（error-handling.md 準拠）。

### 6.1 文言の基本トーン

- **落ち着いた丁寧語**（「〜できませんでした」「〜してください」）
- **原因の説明 + 解決策の提示**（「〜が原因です。〜でお試しください。」）
- **ユーザーを責めない**（「操作が正しくありません」などは使わない）
- **短く、明確に**（一文 40 字以内を目安）

### 6.2 状態別の文言パターン

| 状態 | パターン | 例 |
|------|---------|---|
| ローディング | スケルトンUIで表現（文言は極力出さない） | PostCard スケルトン |
| 空 | 「まだ〜がありません。〜してみましょう。」 | 「まだ投稿がありません。フォローしてみましょう。」|
| エラー | 「〜を読み込めませんでした。」+ 再試行ボタン | 「タイムラインを読み込めませんでした。」|
| オフライン | バナー: 「オフライン中です。接続が回復したら自動的に更新されます。」| 全画面共通バナー |
| 入力エラー | 「〜を入力してください。」/ 「〜は〜文字以内で入力してください。」| インラインエラー |
| 操作エラー(429) | 「〜の上限に達しました。〜またはお試しください。」| 「本日の投稿上限に達しました。明日またお試しください。」|
| 存在しない(404) | 「この〜は見つかりません。」+ 戻るリンク | 「この投稿は見つかりません。」|
| 権限なし(403) | 「この〜を表示する権限がありません。」| 非公開アカウントの投稿リスト |

## 7. ディープリンク方針

Push 通知タップ → 該当画面遷移のルーティングは `lib/push/` の 1 モジュールに集約する。
ディープリンクスキームは `bonlog://` とし、universal links（Android: App Links）は Phase 4 以降に検討。

| 通知種別 | 遷移先 |
|---------|-------|
| いいね | `posts/[postId]` |
| コメント | `posts/[postId]` |
| フォロー | `users/[followerId]` |
| メンション | `posts/[postId]` |
| リポスト | `posts/[postId]` |

未認証でディープリンク到達 → ログイン後に元の画面へ復帰（Expo Router の `redirect` + 遷移先の保持）。
