## 1. 概要 / 目的

ログイン（メール/パスワード）成功後、サーバーが `202 + { requires2FA: true, ticket }` を返した場合に表示するコード入力画面。
ユーザーは認証アプリの TOTP 6 桁コード、またはバックアップコードを入力してログインを完了させる。

**この画面の設計上の肝:** サーバーの ticket は「単回使用・TTL 300 秒」。コード不正でも ticket は消費される。
つまり **1 回でも失敗したら再入力では復帰できず、ログイン画面からやり直す**必要がある。
この制約を画面設計の中核に据え、誤操作と不満を最小化するための UX を提供する。

## 2. ナビゲーション

### 2.1 スタック配置

`(auth)` スタック内のルート `(auth)/two-factor-verify` として配置する。
`navigation-structure.md` §1（app/ 構造）の `(auth)/` 配下に追加する。

```
app/(auth)/
├── login/index.tsx
├── register/...
├── password-reset/...
└── two-factor-verify/
    └── index.tsx          ← 本画面
```

### 2.2 遷移元 / 遷移先

| 区分 | 遷移先 / 遷移元 | 方法 |
|------|----------------|------|
| 遷移元 | `(auth)/login`（ログイン API が 202 + `requires2FA: true` を返したとき）| `router.push()` |
| 遷移先（成功）| `(tabs)/feed` | `router.replace()`（スタックを残さない）|
| 遷移先（失敗・期限切れ）| `(auth)/login` | `router.replace()`（スタックを残さない。後述）|
| 遷移先（戻るボタン）| `(auth)/login` | スタックポップ（Android バックボタン / iOS スワイプバック）|

### 2.3 パラメータ受け渡し

- ログイン画面から ticket 文字列を画面遷移のルートパラメータとして渡す。
- `useLocalSearchParams()` で `ticket` を取得する。型は `string | string[]` のため、文字列であることを型ガードで確認してから使う（`navigation.md` 規約）。
- **注意: ticket はセキュリティ上の秘密情報に準じる。** URL に含まれるため URL エンコードは問題ないが、ログや analytics イベントには ticket の値を記録しない（`auth-tokens.md` に準じる）。
- ticket をメモリ外（AsyncStorage 等）に保存しない。画面パラメータ経由で受け取りそのまま API 呼び出しに使う。

### 2.4 ディープリンク

この画面へのディープリンクは**設けない**。
ticket は一時的な認証状態であり、Push 通知等で外部から遷移するシナリオは存在しない。

## 4. コンポーネント分割

### 4.1 新規コンポーネント: `TwoFactorCodeField`

`components/auth/TwoFactorCodeField` として追加する。

**責務:** TOTP モード / バックアップコードモードの両方に対応する単一テキスト入力フィールド。

**props 概要:**

| prop 名 | 型 | 意味 |
|---------|---|------|
| `mode` | `"totp"` \| `"backup"` | 入力モード（ラベル・キーボード・autoComplete を切り替える） |
| `value` | `string` | 入力値 |
| `onChangeText` | `(text: string) => void` | 変更コールバック |
| `onSubmitEditing` | `() => void` | returnKey タップ時のコールバック（フォーム送信と連動） |
| `hasError` | `boolean` | エラー状態（ボーダー色変化のみ）|
| `disabled` | `boolean` | 送信中の無効化 |
| `ref` | `TextInput ref` | フォーカス制御用 |

**mode による差異:**

| 項目 | totp モード | backup モード |
|------|-----------|------------|
| ラベル | 「認証コード」 | 「バックアップコード」 |
| placeholder | `000000` | `XXXX-XXXXXXXX`（フォーマット例） |
| keyboardType | `"number-pad"` | `"default"` |
| autoComplete | `"one-time-code"` | `"off"` |
| textContentType (iOS) | `"oneTimeCode"` | `"none"` |
| autoCapitalize | `"none"` | `"characters"`（大文字で入力させる）|
| autoCorrect | `false` | `false` |
| maxLength | なし（6 桁確認はサーバー任せ）| なし |

**表示仕様:**

- フィールド高さ: 52pt（TOTP コードを大きめに表示するため auth-forms の標準 48pt より 4pt 増）
- フォント: `textLg`（17pt）/ `letterSpacingWidest`（1.5）/ `textAlign: "center"`
  （6 桁数字を等幅・中央揃えで表示し「コードらしさ」を演出。Web 版の `text-center text-lg tracking-widest` を踏襲）
- バックアップモードでは `textAlign: "left"` に変更（英数字混在のため中央揃えは読みにくい）
- ボーダー・フォーカス・エラースタイルは `auth-forms.md §0.2` の AuthTextField に準じる
- `accessibilityLabel`: totp モード「認証コード入力フィールド」/ backup モード「バックアップコード入力フィールド」

**ペースト対応:**

- TextInput の標準ペースト動作を妨げない（`onChangeText` で受け取る）
- TOTP モードで 6 桁以上の文字列がペーストされた場合、先頭 6 文字のみに切り詰める処理を `onChangeText` 内で行う
  - 根拠: パスワードマネージャーや認証アプリがコードをクリップボードに入れるケースがあり、スペース等を含む場合がある
  - 切り詰めと同時に数字以外を除去する
- バックアップモードではペーストをそのまま受け付ける（ハイフン等のフォーマット文字を含む可能性があるため）

### 4.2 流用コンポーネント（auth-forms.md 定義済み）

| コンポーネント | 用途 |
|-------------|------|
| `AuthPrimaryButton` | 「確認する」ボタン（3 状態：通常 / 無効 / 送信中）|
| `FormErrorMessage` | API エラー・送信失敗のインラインエラー表示 |

## 5. データの流れ

### 5.1 必要なデータ

| データ | 取得元 | 用途 |
|-------|--------|------|
| `ticket` | ルートパラメータ（ログイン画面からの遷移時に付与）| `/api/v1/auth/2fa/verify` の送信ボディ |
| `code` | ユーザー入力（コード入力フィールド）| `/api/v1/auth/2fa/verify` の送信ボディ |

**サーバー API:**

- エンドポイント: `POST /api/v1/auth/2fa/verify`
- リクエスト: `{ ticket: string, code: string }`（code は TOTP 6 桁またはバックアップコード）
- 成功レスポンス（200）: アクセストークン + リフレッシュトークン（ログイン成功と同等の形式。OpenAPI スペックが正）
- エラーレスポンス（401）: `{ code: "AUTH_2FA_INVALID_CODE" | "AUTH_2FA_TICKET_EXPIRED" }`
- レート制限（429）: `{ code: "RATE_LIMITED" }` + `Retry-After` ヘッダー

API 呼び出しは `lib/queries/` のミューテーションフックとして実装する（`architecture.md` 準拠）。

### 5.2 成功後の処理

1. サーバーから受け取ったアクセストークン + リフレッシュトークンを `expo-secure-store` に保存（`auth-tokens.md` 準拠）
2. `queryClient.clear()`（前ユーザーのキャッシュ残留防止）
3. `router.replace(routes.feed)` でフィード画面へ遷移（スタックを残さない）

## 6. 状態とインタラクション

### 6.1 入力モード切替

画面のローカル state として `mode: "totp" | "backup"` を管理する。

- 初期値: `"totp"`
- 「バックアップコードを使用する」タップ → `mode = "backup"` / 入力値をクリア / フィールドへフォーカス
- 「認証アプリのコードを使用する」タップ → `mode = "totp"` / 入力値をクリア / フィールドへフォーカス
- モード切替時に `FormErrorMessage` のエラーもクリアする

**モード切替は API を呼ばない純粋な UI 状態変更**であり、ticket は消費されない。

### 6.2 フォーカス挙動

- 画面表示直後: コード入力フィールドに自動フォーカス（`autoFocus` または `useEffect` で `focus()` 呼び出し）
  - iOS: キーボードが自動的に開く
  - Android: キーボードが自動的に開く（`autoFocus` の動作はプラットフォーム依存のため、実装時に挙動確認を推奨）
- フィールドの `returnKeyType="done"` でキーボードの「完了」タップ → 送信を実行
- `onSubmitEditing` で `handleSubmit` を呼ぶ（空入力の場合は送信しない）

### 6.3 送信ボタンの有効 / 無効

| 状態 | ボタン状態 |
|------|----------|
| コード未入力（空文字） | 無効（disabled）|
| コードあり（1 文字以上）| 有効 |
| 送信中 | 無効 + スピナー表示 |
| エラー後（後述参照）| ボタン非表示 or 無効（エラー種別による。後述）|

### 6.4 送信中の挙動

- `AuthPrimaryButton` をスピナー状態に変更
- `TwoFactorCodeField` を `disabled` にする
- リンク類（切替・戻るリンク）も `disabled` またはポインターイベント無効にする
- キーボードを閉じる（`Keyboard.dismiss()`）

### 6.5 Pull-to-refresh

この画面はデータ取得を行わないため、pull-to-refresh は不要。

## 7. チケット消費の UX（最重要）

### 7.1 基本方針

**「失敗したらログイン画面からやり直し」を明示的に伝え、ユーザーが自力で次のアクションを取れるようにする。**

自動遷移か手動遷移かについては次の原則で判断した:
- チケット期限切れ（`AUTH_2FA_TICKET_EXPIRED`）: **自動遷移しない。手動でログイン画面へ誘導。**
  - 根拠: ユーザーが状況を理解する前に画面が切り替わると、「なぜ戻されたか分からない」混乱が起きる。
- コード不正（`AUTH_2FA_INVALID_CODE`）: **自動遷移しない。エラー表示 → 手動で戻る。**
  - 根拠: 上記と同様。また、ticket がまだ有効な時間内（コード入力直後）に正しいコードを使う可能性が残る場合でも、チケットは消費済みなので再試行しても成功しない。これをユーザーに明示する方が誠実。
- レート制限（`RATE_LIMITED`）: **自動遷移しない。エラー表示 + 戻る導線。**

### 7.2 エラー発生後のフォーム状態

`AUTH_2FA_INVALID_CODE` または `AUTH_2FA_TICKET_EXPIRED` が発生した場合:

1. `FormErrorMessage` にエラーメッセージを表示（エラー種別ごとの文言は §8 参照）
2. `AuthPrimaryButton` を**完全に無効化**（ただし非表示にはしない）
   - `disabled` + opacity 40%（通常の disabled 状態と同じ見た目）
   - 根拠: ボタンが消えると「何もできない」と感じさせる。存在はしているが使えないことを示す
3. コード入力フィールドを `disabled` にする（再入力のムダ打ちを防ぐ）
4. モード切替リンク（「バックアップコードを使用する」等）を**非表示**にする
   - 根拠: ticket 消費後はモード切替しても送信できないため、混乱の種になる
5. `FormErrorMessage` の直下に「ログイン画面へ戻る」ボタンを**強調表示**する（後述）

**「ログイン画面へ戻る」強調ボタンの見た目:**

```
┌─────────────────────────────────────────────────────────┐
│  ✕  [エラーメッセージ（種別ごとの文言）]                  │
│     ログインからやり直してください。                        │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │         ログイン画面へ戻る                          │  │  ← セカンダリボタン
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

このボタンのスタイル:
- 背景: `colorActionSecondary`（`#e9e9e9`）
- テキスト: `colorActionSecondaryText`（`#2e2e2e`）
- 高さ: 48pt / 角丸: `radiusLg`（10pt）
- `accessibilityRole="button"` / `accessibilityLabel="ログイン画面へ戻る"`
- タップ: `router.replace(routes.login)` でログイン画面へ（スタックを積まない）

常時表示している下部の「ログイン画面に戻る」テキストリンクと重複するが、エラー発生後は**このセカンダリボタンをより目立つ位置に追加する**ことで、次のアクションへの誘導を強める。

### 7.3 残り時間（TTL カウントダウン）の扱い

**カウントダウンは表示しない。**

理由:
1. 5 分（300 秒）は一般的な認証アプリの TOTP 画面を開いて入力するには十分な時間。カウントダウンはプレッシャーを与えるだけで UX を向上させない
2. コード不正でも ticket が消費されるため、TTL 内に再入力しようとしても失敗する。「残り時間がある」と表示することで「もう一度試せる」誤解を生む危険がある
3. Web 版（`TwoFactorStep.tsx`）もカウントダウンを表示していない

代わりに、説明文テキストに「コードは一定時間で変わります。最新のコードをご確認ください。」のような補足を入れることで、TOTP の時間制の性質をユーザーに伝える（後述 §8 コピー案参照）。

## 8. エッジケース

### 8.1 ローディング状態

API 送信中の表示は §6.4 で定義。フォームの初期表示は即時（ローディングスケルトン不要）。

### 8.2 空状態

コード未入力でボタンを無効にする。「空」状態の特別な表示は不要。

### 8.3 エラー状態（種別ごと）

| エラーコード / 状況 | `FormErrorMessage` に表示するメッセージ | フォーム状態 |
|-------------------|--------------------------------------|-----------|
| `AUTH_2FA_INVALID_CODE` | 「認証コードが正しくありません。ログインからやり直してください。」（定数名: `ERR_2FA_INVALID_CODE`）| フィールド・ボタン無効化 / 戻るボタン追加表示 |
| `AUTH_2FA_TICKET_EXPIRED` | 「認証の有効期限が切れました。ログインからやり直してください。」（定数名: `ERR_2FA_TICKET_EXPIRED`）| フィールド・ボタン無効化 / 戻るボタン追加表示 |
| `RATE_LIMITED`（429）| 「操作の上限回数に達しました。しばらく待ってから再度ログインしてください。」（定数名: `ERR_2FA_RATE_LIMITED`）| フィールド・ボタン無効化 / 戻るボタン追加表示 |
| ネットワークエラー | 「ネットワーク接続を確認してください。」（定数名: `ERR_NETWORK`）| フィールド・ボタンは**有効のまま**（再試行可能なため）|
| その他サーバーエラー（5xx）| 「認証中にエラーが発生しました。しばらく待ってから再度お試しください。」（定数名: `ERR_2FA_SERVER_ERROR`）| フィールド・ボタンは**有効のまま**（サーバー側の一時的エラーの可能性があるため）|
| `ticket` パラメータ欠落（不正遷移）| 後述 §8.5 参照 | — |

**ネットワークエラーと 5xx はフォームを無効化しない理由:**
ticket は消費されていないため（送信がサーバーに届いていない可能性がある）、再試行を許容する。

### 8.4 オフライン状態

- 送信ボタンタップ時にオフラインを検知した場合: `FormErrorMessage` にオフラインエラーを表示。API 呼び出しはしない
- `OfflineBanner` を画面上部に表示する（`common-states.md §5` 準拠）
- フィールド・ボタンは有効のまま（オンライン復帰後に再試行できるようにする）

### 8.5 ticket パラメータ欠落・不正遷移

`(auth)/two-factor-verify` に ticket なしで直接アクセスした場合（ディープリンク等）:

- 画面マウント直後に `ticket` パラメータを確認する
- 欠落または空文字の場合: `FormErrorMessage` に「認証情報が見つかりません。ログイン画面へ戻ってください。」を表示し、全フォーム要素を無効にする。「ログイン画面へ戻る」ボタンを表示する
- `router.replace()` での自動遷移はしない（ユーザーに状況を伝えてから手動で戻らせる）

### 8.6 権限なし

この画面は `(auth)` スタック内にあり、認証済みユーザーが到達することは認証ガードにより防がれる（`navigation-structure.md §3` 準拠）。

## 11. iOS / Android 差異

| 項目 | iOS | Android |
|------|-----|---------|
| ヘッダーの戻るボタン | 「ログイン」テキスト（前画面タイトル自動継承）+ 「<」| 「←」アイコンボタン |
| バック操作 | ヘッダー戻るボタン / 左端スワイプジェスチャー | ヘッダー戻るボタン / システムバックボタン / ジェスチャー |
| autoFocus + キーボード | キーボードが自動展開（number-pad が即表示）| 同左（動作確認推奨）|
| `textContentType: "oneTimeCode"` | iOS 12 以上で SMS / AirDrop でのコード自動提案が有効 | 効果なし（Android は autoComplete で対応）|
| `autoComplete: "one-time-code"` | 効果なし（iOS は textContentType で対応）| Android で SMS ワンタイムコードの自動入力候補を提示 |
| キーボード | number-pad（TOTP モード）| 同左 |

**keyboardType の挙動に関する補足:**
TOTP モードで `keyboardType="number-pad"` を使うと、Android では「.」「,」等が表示されない純粋な数字キーパッドになる。iOS も同様。バックアップコードモードでは `keyboardType="default"` にして英数字を入力可能にする。

### 12.2 Web 版 `TwoFactorStep.tsx` との整合

| Web 版要素 | モバイル版の対応 |
|-----------|--------------|
| 1 つのフィールドで TOTP / バックアップコードを兼用（`placeholder="000000 または バックアップコード"`）| Web に合わせて 1 フィールドの兼用方式を採用するが、**モバイルはモード切替 UI を追加**する（理由: モバイルの `keyboardType` は TOTP と backup で異なるため、1 フィールドのまま両方に対応するとバックアップ入力時に number-pad が出てしまう）|
| 「確認」ボタン + 「キャンセル」ボタン（Web はキャンセルでログインステップへ戻る） | 「確認する」ボタン + 「ログイン画面に戻る」テキストリンク（モバイルはヘッダーの戻る操作が自然な戻り手段のため、キャンセルボタンは不要。下部にテキストリンクのみ残す）|
| エラー発生後も再入力可能（チケット消費に関する制御なし）| **モバイルは ticket 消費後にフォームを無効化**し、「ログイン画面へ戻る」を強調（モバイルの API 仕様に合わせた差分。Web はサーバーアーキテクチャが異なる）|

| `className="text-center text-lg tracking-widest"` | `textAlign: "center"` / `textLg`（17pt）/ `letterSpacingWidest`（1.5）|

## 13. 未確定事項・要判断

| 項目 | 内容 | 担当 |
|------|------|------|
| `ERR_2FA_*` 定数名の最終確定 | `lib/constants/errors.ts` への追加は frontend の実装時に確定する。本仕様の定数名案はそのまま採用するか、または既存の命名慣行に合わせて調整してよい | **frontend 判断** |
| ネットワークエラー時のリトライ方針 | §8.3 では「フォームを有効のままにする」としたが、ticket に残り時間がある場合のみ再試行を許容するべきか（TTL を画面側で管理するか）。現仕様では TTL 管理を行わない（サーバーに任せる）ため、ネットワーク復帰後に ticket が期限切れになっている可能性がある。ユーザーが再送信して `AUTH_2FA_TICKET_EXPIRED` を受け取る流れを許容するかどうか | **PM 判断** |
| `autoFocus` の Android での挙動 | `autoFocus` prop が Android でキーボードを自動展開する挙動は端末・OS バージョンにより差がある場合がある。実装時に確認し、必要なら `useEffect` + `ref.focus()` に切り替える | **frontend 確認事項** |
| バックアップコードのフォーマット検証 | バックアップコードのフォーマット（文字数・ハイフン有無）はサーバー仕様が正。クライアント側で事前バリデーションするかどうかは OpenAPI スペックのバリデーション定義を確認してから判断する | **core に要確認** |
| ticket の URL エンコード / 安全性 | ticket 文字列をルートパラメータに含めることの安全性（URL ログへの記録等）は、実際のインフラ構成（CDN ログ等）によって判断が変わる。現仕様では許容としているが、インフラ側の URL ロギング設定を確認することを推奨する | **core に要相談** |
