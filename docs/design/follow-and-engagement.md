作成日: 2026-06-14

前提: `design-tokens.md` / `navigation-structure.md` / `post-card.md` / `common-states.md` に準拠
API 出典: `docs/plans/openapi-v1.3.0-snapshot.json` (v1.3.0)

---

## 1. 概要・目的

フォローボタン（プロフィール画面）といいねボタン（PostCard）のインタラクション仕様を確定する。

- フォローは公開 / 非公開アカウントで挙動が分岐し、**未フォロー / 申請中 / フォロー中** の 3 状態を持つ
- いいねは楽観更新で即時フィードバックを返す高頻度操作
- どちらも API が冪等設計のため、楽観更新 + サーバー応答での確定が設計の基本方針

## 2. フォローボタン

### 2.1 3 状態の定義

サーバーは `POST /api/v1/users/{id}/follow` と `DELETE /api/v1/users/{id}/follow` の 2 エンドポイントで状態遷移を管理する。レスポンスの `{ following, requested, followerCount }` を正として状態を更新する。

| 状態名 | サーバー状態 | 意味 |
|--------|------------|------|
| 未フォロー | `following: false, requested: false` | まだフォローしていない（初期状態） |
| 申請中 | `following: false, requested: true` | 非公開アカウントへのリクエストを送信済み |
| フォロー中 | `following: true, requested: false` | フォロー関係が成立している |

`following` と `requested` は同時に `true` にならない（API 仕様による保証）。

### 2.2 ボタンの見た目

#### 未フォロー状態

```
┌─────────────────────┐
│     フォロー        │   ← 塗りつぶしボタン
└─────────────────────┘
```

| 属性 | 値 |
|------|----|
| 背景色 | `colorActionPrimary`（`#2e2e2e` 墨色） |
| テキスト | `colorActionPrimaryText`（`#ffffff`）/ `textBase`（14pt）/ `fontWeight: 600` |
| 枠線 | なし |
| 角丸 | `radiusMd`（8pt）|
| 横幅 | 親コンテナに応じる（プロフィールヘッダーでは `min-width: 120pt`） |
| 縦の高さ | 44pt（タップターゲット最小値） |
| 文言 | 「フォロー」 |

#### 申請中状態

```
┌─────────────────────┐
│     申請中          │   ← アウトライン（枠線）ボタン
└─────────────────────┘
```

| 属性 | 値 |
|------|----|
| 背景色 | `colorBackground`（`#ffffff` 透明に近い白）|
| テキスト | `colorTextPrimary`（`#1a1a1a`）/ `textBase`（14pt）/ `fontWeight: 600` |
| 枠線 | 1.5pt solid `colorBorder`（`#c8c8c8`） |
| 角丸 | `radiusMd`（8pt）|
| 文言 | 「申請中」 |

長押しせずとも「タップで申請を取り消せる」ことをユーザーに気づかせるため、タップ時のフィードバック（次節）が重要。

#### フォロー中状態（通常時）

```
┌─────────────────────┐
│     フォロー中      │   ← アウトライン（枠線）ボタン
└─────────────────────┘
```

| 属性 | 値 |
|------|----|
| 背景色 | `colorBackground`（`#ffffff`）|
| テキスト | `colorTextPrimary`（`#1a1a1a`）/ `textBase`（14pt）/ `fontWeight: 600` |
| 枠線 | 1.5pt solid `colorBorder`（`#c8c8c8`） |
| 角丸 | `radiusMd`（8pt）|
| 文言 | 「フォロー中」 |

#### フォロー中状態（タップ直前 / pressIn 時）

モバイルにはホバー状態がないため、`onPressIn` で「解除」状態への予告フィードバックを出す。

```
┌─────────────────────┐
│   フォローを解除    │   ← テキストと枠線が警告色に変化
└─────────────────────┘
```

| 属性 | 値 |
|------|----|
| 背景色 | `colorErrorBg`（`#fdf0ef`）|
| テキスト | `colorError`（`#c0392b`）/ 同フォント |
| 枠線 | 1.5pt solid `colorError`（`#c0392b`）|
| 文言 | 「フォローを解除」 |

`onPressOut` でボタンを離した場合（指を滑らせてキャンセル）は通常の「フォロー中」表示に戻す。

#### 申請中状態（タップ直前 / pressIn 時）

```
┌─────────────────────┐
│   申請を取り消す   │   ← テキストと枠線が警告色に変化
└─────────────────────┘
```

| 属性 | 値 |
|------|----|
| 背景色 | `colorErrorBg`（`#fdf0ef`）|
| テキスト | `colorError`（`#c0392b`）|
| 枠線 | 1.5pt solid `colorError`（`#c0392b`）|
| 文言 | 「申請を取り消す」 |

### 2.3 確認ダイアログの要否

| 操作 | 確認ダイアログ | 理由 |
|------|--------------|------|
| 未フォロー → フォロー（公開） | **不要** | 即時フォロー確立。間違えてもすぐ解除できる |
| 未フォロー → 申請送信（非公開） | **不要** | 申請はタップで取り消せる。送信コストが低い |
| フォロー中 → フォロー解除 | **不要** | pressIn フィードバックで意図的な操作を担保する。ダイアログは UX 上の摩擦が大きい |
| 申請中 → 申請取り消し | **不要** | 同上 |

Web 版も確認ダイアログなし（`FollowButton.tsx` の実装準拠）。モバイル版でも同方針とする。

pressIn の色変化（§2.2 参照）が「本当に実行するか」を視覚的に問いかける役割を担う。

### 2.4 タップ時の挙動（楽観更新フロー）

#### フォロー（未フォロー → フォロー / 申請）

```
タップ
 → ボタンを disabled にする（連打防止）
 → 楽観更新: 公開アカウントなら「フォロー中」、非公開なら「申請中」に即時切り替え
 → POST /api/v1/users/{id}/follow を呼び出す
 → 成功: レスポンスの following / requested で状態を確定（楽観値の補正）
          followerCount を画面に反映
          ボタンを re-enable
 → 失敗: 楽観更新をロールバックして元の状態に戻す
          エラートーストを表示（§2.6 参照）
          ボタンを re-enable
```

#### フォロー解除 / 申請取り消し（フォロー中 or 申請中 → 未フォロー）

```
タップ（pressIn で予告色を表示 → pressOut 確定）
 → ボタンを disabled にする
 → 楽観更新: 「未フォロー」に即時切り替え
 → DELETE /api/v1/users/{id}/follow を呼び出す
 → 成功: レスポンスで状態確定。followerCount を反映。ボタンを re-enable
 → 失敗: ロールバック。エラートーストを表示。ボタンを re-enable
```

#### ローディング中のボタン表示

disabled 状態でも外観の変化が必要。

| 属性 | 値 |
|------|----|
| 不透明度 | 0.6（現在の状態のスタイルを維持したまま薄くする）|
| インジケータ | ボタン内にスピナー（`ActivityIndicator` / `colorActionPrimaryText` または `colorTextPrimary`）を表示してもよい。ただしボタンサイズが変わらないよう内部で処理する |

### 2.5 自己フォロー不可

`currentUserId === profileUserId` の場合はフォローボタンを**表示しない**。

- 自分のプロフィール（`(tabs)/profile`）: フォローボタンの代わりに「プロフィールを編集」ボタンを表示する（`navigation-structure.md` §4.2 準拠）
- 他ユーザープロフィール（`users/[id]`）で自分の ID と一致した場合: フォローボタン非表示、「プロフィールを編集」ボタンを表示するか、またはプロフィール画面（`(tabs)/profile`）へリダイレクトする

### 2.6 エラー時のトーストとレート制限

| エラーコード | トースト文言（使用定数） | 挙動 |
|------------|----------------------|------|
| ネットワークエラー | 「フォロー操作に失敗しました。もう一度お試しください。」（`ERR_FOLLOW_FAILED`）| 楽観更新をロールバック |
| 429（レート制限） | 「しばらく時間をおいてから再度お試しください。」（`ERR_RATE_LIMIT`）| 楽観更新をロールバック。自動リトライしない |
| 404（相手が存在しない） | 「このユーザーは見つかりません。」（`ERR_USER_NOT_FOUND`）| 楽観更新をロールバック |
| 403（停止 / ゲスト不可） | 「この操作を行う権限がありません。」（`ERR_FORBIDDEN`）| 楽観更新をロールバック |
| 400（自己フォロー）| 表示しない（自己フォロー不可で UI 上タップできないため）| — |

エラー文言はすべて `lib/constants/errors.ts` の定数を使う（インライン文字列禁止 — CLAUDE.md 核心ルール6）。

## 3. いいね楽観更新

### 3.1 タップ時の即時フィードバック

`PostCard` の `PostCardActions` 内のいいねボタン（Heart アイコン）の仕様（`post-card.md` §10 の補完）。

```
タップ
 → 即時に表示を切り替える（楽観更新）:
     未いいね → 済: Heart アイコンを colorError（#c0392b）塗りに変更 + likeCount を +1
     済 → 未: Heart アイコンを colorTextSecondary（#5c5c5c）線のみに変更 + likeCount を -1
 → スケールアニメーション: 1.0 → 1.3 → 1.0（durationFast: 200ms / easingBounce / useNativeDriver: true）
 → POST または DELETE /api/v1/posts/{id}/like を呼び出す
 → 成功: レスポンスの liked / likeCount で値を確定（楽観値とのズレを補正）
 → 失敗: 楽観更新をロールバック（元の liked / likeCount に戻す）+ エラートーストを表示
```

楽観更新中はボタンを disabled にしない（連打への対応は §3.2 参照）。

### 3.2 連打への対応

サーバー側のいいね API はレート制限 `toggle_like`（30/分）がある。

| 状況 | 対応 |
|------|------|
| 素早い連打（人間的な速度） | 最後のタップ状態を優先する **debounce 処理**（待機時間: `LIKE_DEBOUNCE_MS` 定数 / 推奨 300ms）を実装し、API 呼び出し回数を削減する |
| debounce 中のアニメーション | 視覚的には即時に切り替える（ユーザーには滑らかに見せる） |
| 429 受信時 | `ERR_RATE_LIMIT_LIKE` トーストを表示して楽観更新をロールバック。ボタンを短時間 disabled にし、Retry-After ヘッダーの秒数分だけ待機してから re-enable する |

debounce の定数名: `LIKE_DEBOUNCE_MS`。`lib/constants/` に配置（frontend に実装依頼）。

### 3.3 アニメーション仕様

`post-card.md` §10.3 からの引用・補完。

| 項目 | 仕様 |
|------|------|
| 対象 | Heart アイコンのスケール |
| タイミング | `onPress` 確定直後（楽観更新と同時） |
| 値の遷移 | `scale: 1.0 → 1.3 → 1.0` |
| 所要時間 | 合計 200ms（durationFast） |
| イージング | `easingBounce` |
| ドライバ | `useNativeDriver: true`（JS スレッド依存なし） |
| 「戻し」時 | ロールバック時のアニメーションは行わない（状態を即時に戻す） |

### 3.4 未認証ユーザーへの対応

未認証状態でいいねボタンをタップした場合:

- 楽観更新しない（表示を変えない）
- `(auth)/login` 画面へ遷移する（`post-card.md` §10.2 準拠）
- `accessibilityLabel`: 「ログインしていいねする。現在 {N} 件」

### 3.5 オフライン時の対応

- いいねボタンのタップを受け付けるが、API 呼び出しは即座に失敗する
- `ERR_OFFLINE_ACTION` のトーストを表示する
- 楽観更新を行わない（オフライン中は表示状態を変えない）
- 接続が回復してもいいね状態を自動で再送信しない（ユーザーが明示的に操作し直す）

## 4. フォローボタンの配置

### 4.1 他ユーザープロフィール画面（`users/[id]`）

プロフィールヘッダー内に配置する。

```
┌──────────────────────────────────────────────────────────┐
│  [カバー画像（任意）]                                     │
├──────────────────────────────────────────────────────────┤
│  [アバター 64pt]                                          │
│  {ニックネーム}                                           │
│  @{ユーザー識別子}                                        │
│  {自己紹介文}                                             │
│  フォロワー {N}人  フォロー中 {N}人                       │
│                                                          │
│  [フォローボタン]  [メッセージ（MVP スコープ外・将来）]  │
└──────────────────────────────────────────────────────────┘
```

- フォローボタンはプロフィール情報の下、タブの上に配置する
- ボタン幅: `min-width: 120pt`（文言が切れない幅）/ 最大は画面幅の 50%
- `自分のプロフィール` を `users/[id]` で表示した場合（ID が自分と一致）: フォローボタンを非表示にし、「プロフィールを編集」ボタンを表示する

### 4.2 検索結果のユーザー行（`(tabs)/search` 内）

ユーザー検索結果の各行の右端にコンパクトなフォローボタンを配置する。

```
┌──────────────────────────────────────────────────────────┐
│  [アバター 44pt]  {ニックネーム}              [フォロー] │
└──────────────────────────────────────────────────────────┘
```

コンパクト版の仕様:

| 属性 | 値 |
|------|----|
| ボタン幅 | 最小 80pt（文言が「フォローを解除」まで切れないサイズ） |
| 高さ | 36pt（hitSlop で 44pt 以上に補完）|
| 文言 | 状態に合わせて §2.2 と同じ文言を使用。ただし「フォロー」のみで「する」を省略してもよい（コンパクト版）|
| 省略文言 | 「フォロー」/「申請中」/「フォロー中」（pressIn 時は「解除」/「取消」）|

## 5. フォロワー数の表示更新

フォロー操作の成功後、相手プロフィール画面のフォロワー数を反映する。

- レスポンスの `followerCount` を使って即時に表示を更新する（クエリ invalidate と組み合わせる）
- `followers` クエリと `users.detail` クエリを invalidate する（`lib/queries/invalidation-map.md` に反映 — frontend に依頼）

## 6. コンポーネント分割

```
FollowButton                 ← フォローボタン本体（状態管理を含む）
  props:
    targetUserId: string     ← フォロー対象のユーザー ID
    isPublic: boolean        ← 対象が公開アカウントか
    initialFollowing: boolean
    initialRequested: boolean
    currentUserId: string | undefined  ← undefined なら未認証
    size?: 'default' | 'compact'       ← プロフィール用 / 検索行用

LikeButton                   ← PostCardActions 内（post-card.md §10 から分離も可）
  props:
    postId: string
    initialLiked: boolean
    initialLikeCount: number
    currentUserId: string | undefined
```

いずれも `lib/queries/` のミューテーションフックを通じて API を呼び出す（コンポーネント内に直接 API 呼び出しを書かない）。

## 7. アクセシビリティ

### フォローボタン

| 状態 | `accessibilityRole` | `accessibilityLabel` |
|------|-------------------|---------------------|
| 未フォロー | `button` | 「{nickname} をフォローする」 |
| 申請中 | `button` | 「{nickname} へのフォローリクエストを取り消す」 |

| ローディング中 | `button` | 上記 + `accessibilityState: { busy: true }` |

|------|---------------------|

## 8. エッジケース

### フォローボタン

| ケース | 対応 |
|--------|------|
| 相手がアカウントを非公開化 → 公開化（または逆） | プロフィールクエリ再取得時に `isPublic` が更新され、ボタンの挙動が変わる |
| 自分をブロックしている相手 | サーバーが 404 を返す（ブロック有無を秘匿）。「ユーザーが見つかりませんでした」トーストを出してロールバック |
| フォローリクエストが承認された後に画面を見る | プロフィールクエリ再取得で `following: true` になる。`focusManager` 連携の refetch で自動更新 |
| 相手にブロックされた | サーバーが 404 を返す。同上 |

### いいねボタン

| ケース | 対応 |
|--------|------|
| 投稿が削除済み | サーバーが 404 を返す。楽観更新をロールバックし、投稿詳細画面で 404 処理（`navigation-structure.md` §4.3 参照） |
| 楽観更新後にサーバー値と乖離 | レスポンスの `liked` / `likeCount` で上書きして確定値を表示する |

## 10. 既存との一貫性メモ

| Web 要素 | モバイル対応 |
|---------|-----------|
| `FollowButton.tsx` の `getButtonText()` — 「リクエスト済み」 | モバイルでは「申請中」（より短く直感的な表現。Web 版の「リクエスト済み」は PC ホバーで「キャンセル」に変わる設計だが、モバイルは pressIn 方式で代替する）|
| `FollowButton.tsx` の `getButtonText()` — 「フォローリクエスト」 | モバイルでは「フォロー」に統一（タップ後に申請中へ切り替わる流れで十分。説明的な文言は不要） |
| `FollowButton.tsx` の variant: `bonsai`（未フォロー）| `colorActionPrimary`（`#2e2e2e`）塗りつぶしに対応 |
| `FollowButton.tsx` の variant: `outline`（フォロー中 / 申請中）| アウトラインボタン（`colorBorder` 枠線）に対応 |
| `FollowButton.tsx` の `isHovered` ホバー色変更 | モバイルでは `onPressIn` / `onPressOut` による `colorError` / `colorErrorBg` に変換 |
| 確認ダイアログなし（Web 準拠）| モバイルも pressIn フィードバックで代替し、ダイアログなし |
| いいねのスケールアニメーション | `post-card.md` §10.3 と同一。`useNativeDriver: true` |
| `post-card.md` §10 の楽観更新 | 本仕様 §3 で詳細化 |

## 11. 未確定事項・要判断

- **debounce の待機時間（`LIKE_DEBOUNCE_MS`）:** 推奨 300ms だが、実機のタッチレスポンスを計測して調整する。frontend に実装依頼。
- **コンパクトボタンの最小幅:** 「フォローを解除」が 5 文字のため、80pt で収まるか実機確認が必要。収まらない場合は「解除」に短縮する（§4.2 で既に代替文言を定義済み）。
- **フォロワー数の invalidation タイミング:** 楽観更新中は `followerCount` を ±1 して即時反映するか、サーバー応答を待つか。UX 上は楽観更新が望ましいが、フォローリクエスト（非公開）の場合は `followerCount` が変化しないため分岐が必要。**frontend に要確認。**
- **検索画面のフォローボタン:** 検索画面仕様（`search-screen.md`）側でコンポーネント配置を別途確認する。

## 12. §2.6 403 文言の統一について（2026-06-15 更新）

旧バージョンの §2.6 では 403（停止 / ゲスト不可）の文言を「この操作は実行できません。」と記載していたが、`lib/constants/errors.ts` の共有定数 `ERR_FORBIDDEN`（「この操作を行う権限がありません。」）と不一致であった。
