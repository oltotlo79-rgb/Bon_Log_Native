# 確認メール再送の導線仕様 — Bon_Log Native

作成日: 2026-06-23
対象画面:
- `(auth)/register/verify-email-sent`（登録完了後のメール確認待ち画面）← 主要な追加箇所
- `(auth)/login`（ログイン画面 / メール未確認エラー時）← 既存の導線をここで確定
前提: `design-tokens.md` / `auth-forms.md` / `common-states.md` / `navigation-structure.md` に準拠
関連仕様: `auth-forms.md` §3（verify-email-sent 画面）/ §1.5（ログイン画面バリデーション ERR_EMAIL_NOT_VERIFIED）
API（想定）: `POST /api/v1/auth/resend-verification` — 正本は OpenAPI。core に要確認。

---

## 1. 概要・目的

登録メールが届かなかったユーザーが確認メールを再送できる導線を、2 か所に設ける軽量な拡張。
サーバーは列挙攻撃対策として常に 200 を返すため、UI は「常に成功」として扱う。

---

## 2. 導線の配置

### 2.1 導線 A: verify-email-sent 画面への追加（主要）

`(auth)/register/verify-email-sent` 画面に「確認メールを再送する」ボタンを追加する。
この画面は登録完了後に到達する。登録時のメールアドレスをセッション（ナビゲーションパラメータまたはローカル状態）から参照してサーバーへ送る。

### 2.2 導線 B: ログイン画面のメール未確認エラー時（既存確定）

`(auth)/login` で `ERR_EMAIL_NOT_VERIFIED` が返ってきた場合、`FormErrorMessage` の下に「確認メールを再送する」ボタンを表示する（`auth-forms.md` §1.5 および §1.7 に既定されているが、ボタンの仕様が未確定だったため本書で確定する）。
この場合はログインフォームに入力済みのメールアドレスを再送先として使用する。

### 3.3 再送ボタンの仕様

| 属性 | 値 |
|------|---|
| ラベル | 「確認メールを再送する」|
| スタイル | セカンダリボタン（アウトライン）/ 背景 `colorBackground` / 枠線 1.5pt `colorBorder` / 角丸 `radiusLg`（10pt）|
| テキスト | `colorTextPrimary`（`#1a1a1a`）/ `textBase`（14pt）/ fontWeight: 600 / `letterSpacingWidest` |
| 高さ | 48pt（`AuthPrimaryButton` と同一高さで揃える）|
| 幅 | 画面幅 - `spacing4`（16pt）× 2（既存のプライマリボタンと同一幅）|
| 送信中 | テキストを非表示にしてスピナー（`ActivityIndicator` / `colorTextPrimary`）を表示 / disabled |
| クールダウン中 | disabled / opacity: 0.5 / ラベルを「{N}秒後に再送できます」に変更（§3.4 参照）|

セカンダリスタイルを採用する理由: 「ログイン画面へ戻る」がプライマリ（主要アクション）で、再送は補助的な操作のため、視覚的な優先順位を区別する。

### 3.4 クールダウン（連打抑制）の仕様

再送後はサーバーのレート制限（core に要確認）とは別に、クライアント側でも連打を抑制する。

| 項目 | 仕様 |
|------|------|
| クールダウン時間 | 60 秒（`RESEND_EMAIL_COOLDOWN_MS` 定数 / `lib/constants/` に定義）|
| カウントダウン表示 | ボタンラベルを「残り {N} 秒」に変更（秒単位でカウントダウン）|
| カウントダウン実装 | `useEffect` + `setInterval` で 1 秒ごとに更新。アンマウント時にクリア |
| クールダウン完了後 | ボタンを「確認メールを再送する」に戻して再送可能にする |
| クールダウンの永続化 | アプリを閉じてもカウントダウンを維持するかどうかは frontend の実装判断に委ねる。MVP では画面内のローカル状態で管理し、再起動で リセットされることを許容する |

### 3.5 再送後のフィードバック

```
再送ボタンをタップ
  → ボタンをスピナーに変更（disabled）
  → POST /api/v1/auth/resend-verification
    body（想定）: { email: string }
  → 成功（常に 200）:
      成功フィードバックを表示（下記）
      クールダウンを開始（60 秒）
  → 失敗（ネットワークエラー等）:
      エラートーストを表示（§5 参照）
      ボタンを再活性化（クールダウンなし）
```

**成功フィードバックの表示方法:**

メインのヒントテキスト（迷惑メールフォルダ案内）の下に、成功インライン通知を表示する。

```
┌────────────────────────────────────────────────────────┐
│  ✓  確認メールを再送しました。しばらく経ってもメールが  │  ← colorSuccessBg 背景 / 左端 3pt colorSuccess ボーダー
│     届かない場合は、迷惑メールフォルダもご確認ください。│     textSm / colorTextPrimary / radiusMd / padding: spacing3
└────────────────────────────────────────────────────────┘
```

- 成功状態は次の再送をタップするまで表示し続ける（アニメーション消去なし）
- 再送を再度タップした場合は成功メッセージをリセットして再表示する

## 4. 導線 B: ログイン画面のメール未確認エラー時

### 4.1 現状の確認

`auth-forms.md` §1.5 と §1.7 に以下が定義されている:

- `ERR_EMAIL_NOT_VERIFIED` のエラーが返った場合のみ「確認メールを再送する」ボタンを `FormErrorMessage` の下に表示する
- 押すと再送 API を呼び、成功で `verify-email-sent` へ遷移する

### 4.2 ログイン画面の状態仕様（補完）

**エラー表示状態のワイヤーフレーム（メール未確認時）:**

```
  ┌────────────── [フォームカード] ───────────────────────────┐
  │                                                          │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │  ! メールアドレスがまだ確認されていません。確認メール │ │  ← FormErrorMessage（colorErrorBg / colorError ボーダー）
  │  │    のリンクをクリックするか、下のボタンから再送して  │ │
  │  │    ください。                                       │ │
  │  └─────────────────────────────────────────────────────┘ │
  │                                                          │
  │  ┌─────────────────────────────────────────────────────┐ │
  │  │          確認メールを再送する                       │ │  ← 再送ボタン（セカンダリ）
  │  └─────────────────────────────────────────────────────┘ │
  │                                                          │
  │  ...                                                     │
  └──────────────────────────────────────────────────────────┘
```

### 4.3 ログイン画面での再送ボタン仕様

導線 A（verify-email-sent 画面）と同一のスタイルを使用する。
ただし幅はフォームカード内に収まるよう `100%` - フォームカードの内側パディング分。

再送処理フロー:

```
「確認メールを再送する」をタップ
  → ボタンをスピナーに変更（disabled）
  → POST /api/v1/auth/resend-verification（body: フォームに入力済みのメールアドレス）
  → 成功（常に 200）:
      router.push((auth)/register/verify-email-sent) でメール確認待ち画面に遷移
      （成功トーストは出さない。verify-email-sent 画面自体が成功フィードバックになる）
  → 失敗（ネットワークエラー等）:
      エラートーストを表示（§5 参照）
      ボタンを再活性化
```

ログイン画面での遷移先が `verify-email-sent` になる点が導線 A と異なる（導線 A は画面内インライン表示）。

クールダウンはログイン画面では実装しない。サーバーの 429 レート制限に委ねる（画面の役割は「エラー → 再送 → 遷移」という単純なフローのため）。

## 5. エラー処理

### 5.1 エラー定数と文言

| エラー | 使用定数 | 文言 |
|--------|---------|------|
| ネットワークエラー / 再送失敗 | `ERR_RESEND_VERIFICATION_FAILED`（新規）| 「確認メールの再送に失敗しました。もう一度お試しください。」|
| 429（レート制限） | `ERR_RATE_LIMIT` | 「操作が多すぎます。しばらく時間をおいてからお試しください。」|
| 5xx（サーバーエラー） | `ERR_SERVER` | 「サーバーエラーが発生しました。しばらく経ってからお試しください。」|
| オフライン | `ERR_OFFLINE_ACTION` | 「現在オフライン中のため、この操作はできません。接続を確認してください。」|

エラーはいずれもトーストで表示する（インラインエラー表示はしない）。

### 5.2 「常に成功」方針の詳細

セキュリティ上の理由（存在しないメールアドレスの列挙防止）から、サーバーは `POST /api/v1/auth/resend-verification` に対して登録されていないメールアドレスを入力された場合も 200 を返す設計（`auth-forms.md` §4.4 の「セキュリティ上の考慮」と同方針）。

UI は 200 を受け取ったら常に成功表示にする。エラーは 4xx/5xx またはネットワーク失敗の場合のみ表示する。

## 6. コンポーネント分割

### 6.1 新規コンポーネント（コンパクト）

本機能はシンプルなため、新規の大型コンポーネントは作らない。

**`ResendVerificationButton`（`components/auth/ResendVerificationButton.tsx`）**

再送ボタン本体と送信中状態を管理する小型コンポーネント。

| prop | 型 | 説明 |
|------|-----|------|
| `email` | `string` | 再送先メールアドレス |
| `cooldownMs` | `number` | クールダウン時間ミリ秒（デフォルト: `RESEND_EMAIL_COOLDOWN_MS`）|
| `onSuccess` | `(() => void) \| undefined` | 成功時コールバック（ログイン画面での遷移用）|
| `onError` | `(error: unknown) => void` | エラー時コールバック |
| `style` | `ViewStyle \| undefined` | 外部からのスタイル上書き（任意）|

このコンポーネントを verify-email-sent 画面とログイン画面の両方で再利用する。

**クールダウン表示:** `ResendVerificationButton` 内部で管理し、クールダウン中は disabled + ラベル変更で表示する。

### 6.2 既存コンポーネントの利用

- `AuthPrimaryButton` と同じ寸法（高さ 48pt）で、セカンダリスタイル（アウトライン）を使う。`AuthPrimaryButton` を variant で拡張するか、独立したコンポーネントにするかは frontend の実装判断に委ねる。
- `Toast` コンポーネント（既存 `components/common/Toast`）をエラー表示に使用する。
- verify-email-sent での成功インライン表示は `FormErrorMessage` と同一パターン（背景 / ボーダー / テキスト）だが、カラーが `colorSuccessBg` / `colorSuccess` に変わる。`FormErrorMessage` を `variant` props で拡張するか、インラインで実装するかは frontend に委ねる。

| タップターゲット | 高さ 48pt（自然に 44pt 以上を満たす）|

## 10. iOS / Android 差異

本機能は OS 固有 API を使用しないため差異なし。
`ActivityIndicator` の見た目は OS により異なるが、既存の認証フォームコンポーネント（`AuthPrimaryButton`）と同じ挙動のため問題なし。
