## 1. 概要・目的

Google Play ユーザー生成コンテンツポリシーの必須要件として、ユーザーが不快・有害なコンテンツやユーザーに対して**アプリ内から通報・ブロック・ミュートを実行できる導線**を提供する。

Web に機能が存在していてもアプリ内から実行できなければポリシー違反になるため、この仕様は機能要件と同格に扱う（`.claude/rules/store-compliance.md` §UGC 要件）。

### 1.1 解決する問題

- 不適切なコンテンツ・嫌がらせを行うユーザーを安心して申告できる手段が不足していた
- ブロック・ミュートにより不快なユーザーの表示を制御できる手段がモバイルに存在しなかった
- ブロック中・ミュート中ユーザーの一覧と解除導線が設定画面にない状態だった

### 1.2 スコープ

| 機能 | MVP 対象 |
|------|---------|
| 投稿の通報 | 対象 |
| コメントの通報 | 対象 |
| ユーザーの通報 | 対象 |
| ユーザーのブロック / 解除 | 対象 |
| ユーザーのミュート / 解除 | 対象 |
| ブロックリスト画面 | 対象 |
| ミュートリスト画面 | 対象 |

## 2. エントリポイントとメニュー設計

### 2.1 メニュー UI の基本方針（iOS / Android 共通）

「3点メニュー（⋮）」のタップでコンテキストメニューを表示する。**ボトムシート（ActionSheet 形式）** を使う。

```
iOS   : ActionSheetIOS（ネイティブ）
Android: Modal + リスト UI（カスタム）
```

Android カスタムシートの外形仕様:

```
背景    : colorSurface (#fcfcfc)
角丸    : radius2xl (18pt) ※上端のみ
影      : shadow-washi-lg
ハンドル: 幅 36pt / 高さ 4pt / colorBorderLight / 上端から spacing2
```

各メニュー項目の仕様:

```
高さ      : 56pt（タップターゲット 44pt 以上確保）
padding 水平: spacing5
アイコン  : 20pt / 左端
テキスト  : textMd (15pt) / colorTextPrimary
セパレータ: 1pt / colorBorderLight
```

破壊的アクション（通報・ブロック）は `colorError (#c0392b)` でテキスト・アイコンを表示する。

### 2.2 投稿カード「⋮」メニュー（PostCardHeader）

既存の `post-card.md` §13 を拡張・具体化する。

**表示条件による項目の切り替え:**

| 閲覧者 | 投稿者との関係 | 表示項目 |
|--------|--------------|---------|
| ログイン済み（自分の投稿） | `currentUserId === post.user.id` | 「編集」「プロフィールに固定」「削除」 |
| ログイン済み（他人の投稿） | `currentUserId !== post.user.id` | 「この投稿を非表示」「投稿を通報」「ユーザーをブロック」「ユーザーをミュート」 |
| 未認証 | — | ⋮ボタン自体を非表示にする |

**他人の投稿メニューのワイヤーフレーム:**

```
┌──────────────────────────────────────────┐
│ ▔▔▔▔▔▔▔▔▔▔（ハンドル）                  │
│ [EyeOff]  この投稿を非表示              │  colorTextPrimary
│ ─────────────────────────────────────── │
│ [Flag]    投稿を通報                    │  colorError
│ [Ban]     @{nickname} をブロック        │  colorError
│ [VolumeX] @{nickname} をミュート        │  colorError
│ ─────────────────────────────────────── │
│ [X]       キャンセル                    │  colorTextSecondary
└──────────────────────────────────────────┘
```

**ミュート状態・ブロック状態での項目変化:**

サーバーが `UserProfileResponse` に `isBlocked` / `isMuted` フラグを返す想定（正本は OpenAPI。**core に要確認**）。

- `isMuted === true` の場合: 「ミュート」を「ミュートを解除」に変更する
- ブロック中のユーザーの投稿はサーバー側でフィード除外済みのため、フィード内での「ブロック中」メニュー表示は原則発生しない。投稿詳細で `isBlocked === true` の場合は「ブロックを解除」を表示する

### 2.3 プロフィール画面「⋮」メニュー（`users/[id]`）

他ユーザープロフィールのヘッダー右上に `⋮` ボタン（44pt × 44pt）を配置する。自分のプロフィールには表示しない。

**フォローボタンとの位置関係:**

```
┌──────────────────────────────────────────────────────────┐
│  [アバター 64pt]                                          │
│  {ニックネーム}                                           │
│  @{ユーザー識別子}                                        │
│  {自己紹介文}                                             │
│  フォロワー {N}人  フォロー中 {N}人                       │
│                                                          │
│  [フォローボタン（flex: 1）]  [⋮ メニューボタン（44pt）] │
└──────────────────────────────────────────────────────────┘
```

`⋮` ボタン: 正方形 44pt / アウトラインボタン / `colorBorder` 枠線 1.5pt / `radiusMd`

**プロフィールメニューの項目（`isBlocked` / `isMuted` で動的に変化）:**

| `isBlocked` | `isMuted` | メニュー項目 |
|-------------|-----------|------------|
| false | false | 「ユーザーを通報」「ブロック」「ミュート」 |
| false | true | 「ユーザーを通報」「ブロック」「ミュートを解除」 |
| true | — | 「ユーザーを通報」「ブロックを解除」 |

ブロック中の場合はミュート項目を非表示にする（ブロックがミュートの上位概念のため）。

**未ブロック・未ミュート状態のワイヤーフレーム:**

```
┌──────────────────────────────────────────┐
│ ▔▔▔▔▔▔▔▔▔▔（ハンドル）                  │
│ [Flag]    @{nickname} を通報             │  colorError
│ [Ban]     @{nickname} をブロック         │  colorError
│ [VolumeX] @{nickname} をミュート         │  colorError
│ ─────────────────────────────────────── │
│ [X]       キャンセル                    │  colorTextSecondary
└──────────────────────────────────────────┘
```

## 3. ブロックフロー

### 3.1 ブロック確認ダイアログ（必須）

ブロックは**相互フォローの自動解除**という不可逆な影響を伴うため、確認ダイアログを必ず表示する。

**ダイアログ仕様:**

```
┌──────────────────────────────────────────┐
│  @{nickname} をブロックしますか？         │  textLg / colorTextPrimary
│                                          │
│  ブロックすると、互いにフォローして      │  textBase / colorTextSecondary
│  いる場合はフォロー関係が解除され、      │
│  相手はあなたのプロフィールを検索・      │
│  フォローできなくなります。              │
│                                          │
│  [キャンセル]       [ブロックする]       │
└──────────────────────────────────────────┘
```

| ボタン | スタイル | 挙動 |
|--------|---------|------|
| 「キャンセル」 | テキストボタン / `colorTextSecondary` | ダイアログを閉じる |
| 「ブロックする」 | 塗りつぶしボタン / 背景 `colorError` / テキスト `colorTextInverse` / 高さ 44pt | §3.2 のブロック処理を実行 |

**iOS / Android 差異:**
- iOS: `Alert.alert()` のネイティブダイアログ（destructive アクションは OS が赤色で表示）
- Android: カスタムモーダルダイアログ（`colorError` 背景のボタンを使用）

**アクセシビリティ:**
- ダイアログ表示時にフォーカスをタイトルに移動させる
- `accessibilityRole="alert"` をダイアログコンテナに付与

---

### 3.2 ブロック送信フロー

```
確認ダイアログで「ブロックする」をタップ
  → ダイアログの「ブロックする」ボタンを disabled にする（スピナー表示）
  → POST /api/v1/users/{id}/block を呼び出す（想定エンドポイント。正本は OpenAPI）
  → 成功:
      ダイアログを閉じる
      成功トーストを表示:「@{nickname} をブロックしました」
      §5 の invalidation を実行
      遷移:
        投稿カードから操作した場合 → フィードへ戻り、投稿カードが消える（invalidate で再フェッチ）
        プロフィール画面から操作した場合 → §3.4 のブロック後表示に切り替える
  → 失敗:
      ダイアログのボタンを re-enable する
      エラートーストを表示（§3.5 参照）
      ダイアログは閉じない（ユーザーが再試行できる状態を維持）
```

---

### 3.3 ブロック解除フロー

ブロック解除は設定画面（`settings/blocked`）またはプロフィール画面のメニューから実行できる。

**確認ダイアログ:** 不要（ブロック解除はフォロー関係が復活しない限定的な影響のため）。

```
「ブロックを解除」をタップ
  → ボタンを disabled にする
  → DELETE /api/v1/users/{id}/block を呼び出す（想定エンドポイント。正本は OpenAPI）
  → 成功:
      成功トーストを表示:「@{nickname} のブロックを解除しました」
      §5 の invalidation を実行
      ブロックリスト（settings/blocked）から操作した場合 → リストアイテムを削除
      プロフィール画面から操作した場合 → メニューを「ブロック」に戻す
  → 失敗:
      ボタンを re-enable する
      エラートーストを表示（§3.5 参照）
```

---

### 3.4 ブロック後のプロフィール画面表示

`users/[id]` でブロック成功後にそのまま画面に留まる場合のレイアウト:

```
┌──────────────────────────────────────────────────────────┐
│  [アバター 64pt]                                          │
│  {ニックネーム}                                           │
│  @{ユーザー識別子}                                        │
│                                                          │
│  [ブロックを解除（colorActionSecondary）]  [⋮]           │
│  ─────────────────────────────────────────────────────── │
│  「このユーザーをブロックしています」                      │
│   colorTextSecondary / textBase / 中央寄せ               │
│                                                          │
│  （投稿リストは非表示）                                   │
└──────────────────────────────────────────────────────────┘
```

- 自己紹介・フォロワー数は省略する（ブロック中は閲覧できないこととする）
- 「ブロックを解除」ボタンはフォローボタンの位置に配置する

**ブロック後の画面遷移方針（未確定）:** ブロック直後に `users/[id]` に留まるか、前の画面に戻るかは PM の判断が必要（§16 参照）。

---

### 3.5 エラー処理（ブロック / ブロック解除）

| エラー | 使用定数 | 文言 |
|--------|---------|------|
| ネットワークエラー / ブロック | `ERR_BLOCK_FAILED`（既存 264行目） | 「ブロックに失敗しました。」 |
| ネットワークエラー / ブロック解除 | `ERR_UNBLOCK_FAILED`（既存 267行目） | 「ブロック解除に失敗しました。」 |
| 404（相手が存在しない） | `ERR_USER_NOT_FOUND`（既存 136行目） | 「このユーザーは見つかりません。」 |
| 429（レート制限） | `ERR_RATE_LIMIT`（既存 114行目） | 「操作が多すぎます。しばらく時間をおいてからお試しください。」 |
| 403（権限なし） | `ERR_FORBIDDEN`（既存 139行目） | 「この操作を行う権限がありません。」 |
| 409（既にブロック済み） | `ERR_BLOCK_ALREADY_BLOCKED`（新規 — §13） | 「すでにブロックしています。」 |
| オフライン | `ERR_OFFLINE_ACTION`（既存 22行目） | 「現在オフライン中のため、この操作はできません。接続を確認してください。」 |

## 4. ミュートフロー

### 4.1 確認ダイアログの要否

ミュートはフォロー関係を解除しない軽量な操作であり、誤操作しても容易に取り消せる。**確認ダイアログは表示しない**。タップ後、直ちに API を呼び出す。

---

### 4.2 ミュート送信フロー

```
「ミュート」をタップ（メニューから）
  → メニューを閉じる
  → POST /api/v1/users/{id}/mute を呼び出す（想定エンドポイント。正本は OpenAPI）
  → 成功:
      成功トーストを表示:「@{nickname} をミュートしました」
      §5 の invalidation を実行
  → 失敗:
      エラートーストを表示（§4.4 参照）
```

---

### 4.3 ミュート解除フロー

```
「ミュートを解除」をタップ（メニューまたは設定画面から）
  → DELETE /api/v1/users/{id}/mute を呼び出す（想定エンドポイント。正本は OpenAPI）
  → 成功:
      成功トーストを表示:「@{nickname} のミュートを解除しました」
      §5 の invalidation を実行
  → 失敗:
      エラートーストを表示（§4.4 参照）
```

---

### 4.4 エラー処理（ミュート / ミュート解除）

| エラー | 使用定数 | 文言 |
|--------|---------|------|
| ネットワークエラー / ミュート | `ERR_MUTE_FAILED`（新規 — §13） | 「ミュートに失敗しました。もう一度お試しください。」 |

## 5. ブロック・ミュート成功後の即時反映（invalidation 方針）

サーバー側でブロック・ミュートユーザーはフィード・検索・通知から除外済みのため、クライアントの in-memory フィルタは不要。`invalidateQueries` でデータを再フェッチすることで除外済みデータが自動的に反映される。

### 5.1 ブロック成功後に invalidate するクエリキー

| クエリキー（想定） | 理由 |
|------------------|------|
| `posts.feed` | フィードからブロック相手の投稿を除去する |
| `users.detail(blockedUserId)` | プロフィールの `isBlocked` フラグを更新する |
| `users.followers` / `users.following` | フォロー関係の自動解除を反映する |
| `notifications` | ブロック相手の通知を除去する |
| `users.blocks` | 設定画面のブロックリストに追加する |
| `search.users` | 検索結果からブロック相手を除去する |

### 5.2 ミュート成功後に invalidate するクエリキー

| クエリキー（想定） | 理由 |
|------------------|------|
| `posts.feed` | フィードからミュート相手の投稿を除去する |
| `users.detail(mutedUserId)` | プロフィールの `isMuted` フラグを更新する |
| `notifications` | ミュート相手の通知を除去する |
| `users.mutes` | 設定画面のミュートリストに追加する |

クエリキーは `lib/queries/keys.ts` の命名に準拠する。invalidation 対応表 `lib/queries/invalidation-map.md` への追記を frontend に依頼する。

## 7. 通報フロー

### 7.1 通報対象種別

| 種別 | エントリポイント | `targetType` 値 |
|------|--------------|---------------|
| 投稿 | 投稿カード「⋮」→「投稿を通報」 | `post` |
| コメント | コメント「⋮」→「コメントを通報」 | `comment` |

### 7.5 送信処理

```
「通報する」をタップ
  → ボタンを disabled にする（スピナー表示）
  → POST /api/v1/reports を呼び出す（想定エンドポイント。正本は OpenAPI）
    リクエスト body（想定）:
      targetType: 'post' | 'comment' | 'user'
      targetId: string
      reason: 'spam' | 'inappropriate' | 'harassment' | 'copyright' | 'other'
      detail?: string   ← 最大 1000 文字
  → 成功:
      モーダルを閉じる

```

## 10. データの流れ（API 依存）

以下は**想定エンドポイント**。正本は OpenAPI スペック（Bon_Log_cfw 管轄）。

| 操作 | メソッド | 想定パス |
|------|---------|---------|

| ブロック解除 | `DELETE` | `/api/v1/users/{id}/block` |

| ミュートリスト取得 | `GET` | `/api/v1/users/me/mutes` |

**`UserProfileResponse` への追加フィールド（cfw に要確認 — 最優先）:**

- `isBlocked: boolean` — 閲覧者が対象ユーザーをブロックしているか
- `isMuted: boolean` — 閲覧者が対象ユーザーをミュートしているか

これらのフラグがない場合 N+1 呼び出しが発生するため、`UserProfileResponse` への埋め込みをサーバー側に依頼すること（**core に要相談**）。

**`PostResponse` 内 `user` オブジェクトへの `isBlocked` / `isMuted` 埋め込みも core に要相談。**

## 14. `follow-and-engagement.md` §2.6 文言統一の決定記録

前回評価指摘の対応として、`follow-and-engagement.md` §2.6 の 403 エラー文言「この操作は実行できません。」を `ERR_FORBIDDEN`（「この操作を行う権限がありません。」）に統一した。

**決定理由:**

1. CLAUDE.md 核心ルール6「エラーメッセージは `lib/constants/errors.ts` の定数を使用」に従い、インライン文字列を使わない
2. フォロー操作専用の 403 文言を新設するほどの意味的差異がなく、`ERR_FORBIDDEN` で十分に意図が伝わる
3. 全画面で同じ 403 エラーに同じ文言を使う方が、ユーザーの一貫した体験につながる

**変更済みファイル:**
- `docs/design/follow-and-engagement.md` §2.6 の表の 403 行を更新
- §12 に決定内容のメモを追記
- ヘッダーに「更新日: 2026-06-15」を追記
