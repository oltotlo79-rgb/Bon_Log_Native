# Push 通知ルール

## 構成（計画書 工事#4）

- 受信: expo-notifications + FCM / APNs
- 送信はサーバーの `lib/services/notification-core`（Bon_Log_cfw 管轄）。通知の発火条件・本文はサーバーが決める
- クライアントの責務は**許可取得・デバイストークン登録・受信表示・タップ遷移のみ**

## デバイストークン

- 登録: 通知許可が取れた後に、サーバーのデバイストークン登録 API（`/api/v1/devices` 系。OpenAPI スペックが正）へ登録する
- 失効: ログアウト時にサーバー側の登録を解除し、ローカルの登録状態もリセットする（`auth-tokens.md` のログアウト手順に含まれる）
- Expo push token の変化を検知したら再登録する
- デバイストークンは秘匿情報に準じる — ログに出さない

## 許可リクエスト

- **初回起動で即座に要求しない**。ログイン後、または通知に価値が生まれる文脈（フォロー・いいねの後等）で要求する（OS の許可ダイアログは実質一度きりの機会）
- 拒否済みの場合は OS 設定アプリへの導線を出す（ダイアログの再表示はできない）

## 受信ハンドリング

- フォアグラウンド受信時の表示方針（バナー表示 / 抑制）は NotificationHandler で一元定義する
- タップ → ペイロード内の遷移情報 → `lib/push/` の対応表で Expo Router の遷移へ変換（`navigation.md`）
- 通知一覧画面のデータは通常の API から取得する。**Push はあくまで到達手段であり、通知データの正はサーバー API**
- Android は通知チャネル（重要度・サウンド）を定義する。チャネル ID は `lib/constants/` の定数
